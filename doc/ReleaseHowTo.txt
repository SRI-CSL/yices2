HOW TO PREPARE A YICES RELEASE

This checklist describes repository preparation. The authoritative public
downloads are the GitHub Release assets.

1. Select and validate the release commit.

   - Work on the preparing-release-2.8.0 branch.
   - Ensure the worktree is clean and all intended pull requests are merged.
   - Reconcile the complete diff from the previous release tag with the
     milestone and release notes.

2. Update the version number.

   The release version must agree in:

   - Makefile
   - src/include/yices.h
   - doc/manual/manual.tex
   - doc/sphinx/source/conf.py
   - doc/yices.1
   - doc/yices-sat.1
   - doc/yices-smt.1
   - doc/yices-smt2.1
   - debian/changelog
   - debian/control

3. Validate licensing.

   - LICENSE.txt must contain the BSD 3-Clause License.
   - SRI-owned source files must carry the approved copyright/SPDX notice.
   - Third-party notices and historical artifacts must retain their original
     terms.
   - Source and binary distributions must include LICENSE.

4. Build and test from a clean checkout.

   Run the release, development, API, unit, regression, MCSAT, thread-safe,
   and delegate configurations required by the release checklist. Build the
   Sphinx documentation and PDF manual.

5. Build distributions from the validated release commit.

   Source:
     make source-distribution

   Linux x86-64 static GMP:
     make static-distribution

   macOS Intel static GMP:
     make static-distribution

   macOS Apple Silicon static GMP:
     make static-distribution

   Record the exact asset filenames and compute checksums.

6. Tag and publish on GitHub.

   - Create the lightweight tag yices-2.8.0.
   - Push the tag.
   - Create the GitHub Release from that tag.
   - Upload the verified platform assets and checksums.
   - Download and smoke-test every published artifact.

7. Update the Yices website.

   - Check out the exact release tag in the adjacent yices2 repository.
   - Update Portals/yices.csl.sri.com release metadata and notes.
   - Use the exact GitHub asset URLs; do not guess filenames.
   - Run:
       ant clean refresh-manual check-release
   - Publish only after the generated pages and links have been reviewed.

8. Complete downstream publication.

   - Update Homebrew and Debian/Ubuntu PPA packaging.
   - Close the GitHub milestone.
   - Publish the release announcement.
   - Perform final public website and download smoke tests.
